# webapp-i18n: Webapp i18n: ru/en driven by bot user.lang

## Context

Вся вёрстка `webapp/templates/*.html` захардкожена на русском. Пользователи
с английской локалью в боте (`MyMtlWalletBotUsers.lang = 'en'`) получают
смешанный UI: бот на английском, миниапп на русском. Нужно привести
миниапп в соответствие с выбранным в боте языком.

### Ключевые решения (подтверждены в брейнштормингa)

1. **Источник языка — БД бота** (`user.lang`), не `Telegram.WebApp.initDataUnsafe.user.language_code`. Telegram UI lang может
   расходиться с явным выбором пользователя в боте.
2. **Передача во фронт — через URL query-параметр** `?lang=ru`/`?lang=en`,
   который бот добавляет при построении `WebAppInfo.url` в
   `bot/keyboards/webapp.py`. Webapp к БД не ходит — Redis остаётся
   единственным shared state.
3. **Каталог строк — один файл** `webapp/static/js/i18n.js` с `const I18N =
   { ru: {...}, en: {...} }` и helper `t(key, params)`. Шаблоны подключают
   этот файл и используют атрибуты `data-i18n="page.element"` для
   статического текста + `t(...)` для динамических строк в JS.
4. **Охват — все четыре шаблона:** `base.html`, `sign.html`, `import.html`,
   `settings.html`. Иначе юзер с `ru` откроет импорт и увидит смесь языков.
5. **Fallback:** неизвестный/отсутствующий `?lang=` → `en`. Отсутствующий
   ключ в активном языке → возврат самого key + `console.warn` (моментально
   видно пропуск при разработке).

## Files/Directories To Change

Бот:
- `bot/keyboards/webapp.py` — обе функции (`webapp_sign_keyboard`,
  `webapp_import_key_keyboard`) добавляют `&lang={lang}` в URL. Язык берут
  через `app_context.localization_service.get_user_language(user_id)`
  (sync, cache-based, fallback `'en'`).
- `bot/tests/test_signing_flow.py` — расширить существующие тесты
  `test_webapp_sign_keyboard_structure` /
  `test_webapp_import_key_keyboard_structure` проверкой `?lang=ru|en` в URL.

Webapp:
- `webapp/static/js/i18n.js` (новый) — каталог `I18N`, функции
  `getLangFromURL()`, `t(key, params)`, `applyTranslations()`.
- `webapp/templates/base.html` — подключает `<script
  src="{{ url_for('static', path='js/i18n.js') }}">` до остальных скриптов,
  вызывает инициализацию lang до применения к DOM.
- `webapp/templates/sign.html`, `import.html`, `settings.html` —
  размечают статический текст через `data-i18n`, заменяют русские литералы
  в JS-вызовах на `t("sign.xdr_copied")` и аналоги.
- `webapp/app.py` — backend не трогаем. Вся i18n-логика на клиенте.

## Architecture / Data Flow

```
MyMtlWalletBotUsers.lang (DB)
        │
        ▼
localization_service.user_lang_cache  (in-memory, pre-warmed by middleware)
        │
        ▼
bot/keyboards/webapp.py → WebAppInfo(url=".../sign?tx=...&lang=ru")
        │
        ▼
Telegram client opens webapp
        │
        ▼
webapp base.html → i18n.js::getLangFromURL() → window.__LANG = "ru"
        │
        ▼
applyTranslations() walks [data-i18n*] → replaces text
        │
        ▼
Runtime JS → t("sign.xdr_copied") → localized toast
```

## String catalog conventions

- Namespace: `<page>.<element>`. Плоский, без вложенностей.
- Ключи лежат в `i18n.js`, примеры:
  - `common.cancel`, `common.close`, `common.confirm`
  - `sign.card_title`, `sign.btn_sign`, `sign.btn_sign_progress`,
    `sign.enter_hint`, `sign.password_title`, `sign.password_hint`,
    `sign.xdr_copied`, `sign.already_signed`, `sign.layout_warning`
  - `import.title`, `import.btn_save`, ...
  - `settings.title`, ...
- Обе версии (ru/en) обязательны для каждого ключа.

## Risks / Open Questions

- **`settings.html` / `import.html` контент:** нужно собрать полный
  список строк и утвердить переводы. Возможно, часть терминов требует
  уточнения у пользователя (биометрия, ключевые фразы).
- **`data-i18n-html`:** нужен ли вариант для строк, содержащих inline-иконки
  (например, галочка ✓). Пока оставляем минимум: `textContent` +
  `placeholder`.
- **Перезапуск приложения:** при смене `user.lang` в боте через `/settings`
  старые WebApp-кнопки (уже отправленные в чат) содержат старый `lang` в
  URL. Это приемлемо — после следующего `/sign` URL обновится. Делать
  backfill не нужно.
- **FIELD_LANG в Redis:** на будущее (если webapp понадобится язык для
  страниц, открываемых без бот-кнопки) можно рассмотреть, но в scope этой
  задачи — нет.
